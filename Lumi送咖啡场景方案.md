# 机器人送咖啡
## 需求背景
送咖啡机器人主要服务对象为法院内的工作人员。由法院工作人员在线下单后，Lumi（机器人）自动前往取餐点拿取咖啡。根据订单中的房间信息，前往相应房间，将咖啡放置在指定位置。

## 实现方案：
- 法院1楼的咖啡店为取餐点
- 工作人员直接在咖啡店的线上平台下单咖啡
- Lumi在咖啡店指定位置等候
- 咖啡店员将制作完成的咖啡放入Lumi的托盘中，并在App/触摸屏（未确定）上确认订单送往的房间信息
- Lumi在店员确认好后，开送配送
- Lumi需连接法院的梯控系统呼叫电梯，并下达电梯前往指定楼层
- 到达指定楼层后，Lumi前往相应的房间门口（各层地图信息已预先创建）
- 将咖啡放置在房间门口的小桌上。此动作对精度要求不高，若小桌固定，可直接依赖AGV底盘自身精度完成。也可在放置前先通过视觉重新校准放置位置
- 完成配送后，Lumi再次回到取餐点

## 异常处理机制

### 电梯相关异常

| 异常场景 | 检测方式 | 应对机制 |
|---|---|---|
| 电梯呼叫超时（如超过60s无响应） | 梯控系统返回超时 | 重试呼叫最多3次，每次间隔30s；3次失败后暂停该订单，向后台发送告警，由人工介入处理 |
| 电梯运行中故障/停运 | 梯控系统推送故障状态 | Lumi保持当前状态不动，立即上报后台；待电梯恢复后自动继续，或由调度系统将订单标记为"待人工处理" |
| 到达目标楼层后电梯门未正常开启 | 梯控返回门状态异常 | 等待10s后重新发送开门指令，最多重试2次；失败则上报后台 |
| 电梯内人员满载，Lumi无法进入 | 激光雷达/视觉检测电梯内空间不足 | Lumi等候下一班电梯，最多等候3班；仍无法进入则上报后台 |

### 网络与通讯异常

| 异常场景 | 检测方式 | 应对机制 |
|---|---|---|
| 与梯控系统通讯断连 | 心跳超时（如10s无响应） | Lumi原地停驻，进入安全等待模式；尝试重连，重连成功后恢复任务；重连超时（如2min）则上报后台 |
| 与订单调度服务断连 | 心跳超时 | Lumi继续完成当前正在执行的配送任务（已下发的目标楼层和房间信息缓存在本地），完成后在取餐点待命，不再接受新订单 |
| Wi-Fi信号弱/不稳定 | 信号强度低于阈值 | Lumi减速行驶，在关键节点（如进出电梯前）主动确认连接状态；若断连则按上述断连策略处理 |

### 导航与运动异常

| 异常场景 | 检测方式 | 应对机制 |
|---|---|---|
| 房间门口有障碍物阻挡 | 激光雷达/视觉检测到障碍物 | 尝试绕行（重新规划局部路径）；若无法绕行，等待60s后重新检测；3次尝试后仍无法到达，上报后台并返回取餐点 |
| 行驶路径被完全封堵 | 导航模块返回路径不可达 | 等待120s后重新规划路径；若仍不可达，暂停订单并上报后台 |
| 定位丢失/漂移 | 定位模块置信度低于阈值 | Lumi立即停止移动，尝试通过视觉/激光重新定位；重定位失败则原地待命并上报后台 |
| 底盘运动异常（如卡住、打滑） | 底盘IMU/编码器数据异常 | 立即停止电机输出，上报异常状态，等待人工介入 |

### 放置与配送异常

| 异常场景 | 检测方式 | 应对机制 |
|---|---|---|
| 视觉校准失败（无法识别放置位置） | 视觉模块未检测到目标标记 | 回退到底盘自身精度放置（盲放模式）；若盲放也失败，保持咖啡在托盘内，返回取餐点并上报 |
| 咖啡在配送途中倾倒/滑落 | 托盘传感器/视觉检测到异常 | 立即停止移动，上报后台；由人工判断是否继续配送或取消订单 |
| 目标房间信息无法识别 | 订单数据解析失败 | 在取餐点待命，向后台发送告警，等待人工修正订单信息 |

### 设备状态异常

| 异常场景 | 检测方式 | 应对机制 |
|---|---|---|
| 电量低于阈值（如20%） | BMS上报电量 | 完成当前配送后立即返回充电点；若电量低于10%，立即中止任务返回充电 |
| 硬件模块故障（如传感器、电机） | 模块自检异常 | 立即停止任务，进入安全停车模式，上报后台请求人工维护 |

### 异常上报机制

- 所有异常事件均记录到日志系统，包含时间戳、位置、异常类型、处理结果
- 关键异常（电梯故障、通讯断连、硬件故障）实时推送至运维人员终端
- Lumi本地缓存最近一条任务的完整状态，确保断连恢复后可继续执行

## 多订单并发调度与排队策略

### 基本调度模型

Lumi采用**单机器人串行配送**模式，即一次只执行一个配送任务，任务按队列顺序处理。

### 任务队列管理

- **订单状态流转**：`下单 → 制作中 → 已出餐 → 排队等待 → 配送中 → 已送达 → 完成`
- Lumi仅接收状态为"已出餐"的订单进入配送队列
- 队列采用 **FIFO（先进先出）** 为默认排序规则

### 优先级规则

| 优先级（高→低） | 规则 |
|---|---|
| 1 | 人工手动插队/加急标记的订单 |
| 2 | 等待时间超过阈值（如15min）的订单自动提权 |
| 3 | 同楼层订单优先（减少电梯往返次数） |
| 4 | 按下单时间排序（FIFO） |

### 同楼层批量配送优化

- 当队列中存在多个目标楼层相同的订单时，调度系统将其**合并为一个批次任务**
- Lumi一次乘梯前往该楼层，按房间顺序依次配送，全部完成后返回取餐点
- 同楼层批量配送期间，新到达的该楼层订单可追加到当前批次（前提：Lumi尚未离开该楼层）

### 任务取消与中断

- 订单可在"排队等待"状态前被取消，取消后自动从队列移除
- 配送中的订单不可取消（Lumi已出发），但可由后台标记为"中止"，Lumi将携带咖啡返回取餐点
- Lumi从取餐点出发后，若收到更高优先级的紧急任务，可在当前任务完成后优先处理

### 调度服务接口（规划）

| 接口 | 说明 |
|---|---|
| `POST /order/ready` | 店员确认出餐，订单进入配送队列 |
| `GET /queue/status` | 查询当前配送队列状态 |
| `POST /queue/cancel/{orderId}` | 取消指定订单 |
| `POST /queue/prioritize/{orderId}` | 手动提升订单优先级 |
| `GET /lumi/status` | 查询Lumi当前状态（位置、电量、当前任务） |

## 软件开发工作

整个软件系统分为三大部分：**人机交互系统**、**机器人主程序**和**设备间通讯模块**。

---

### 一、人机交互开发

包含交互界面与对应服务器两部分，共同构成订单调度中枢。

#### 1.1 交互界面

| 功能模块 | 说明 |
|---|---|
| 订单确认页 | 店员在出餐后选择目标房间信息，确认发起配送 |
| 队列状态页 | 实时展示当前待配送订单列表、Lumi位置及状态 |
| 异常处理页 | 显示异常告警，支持人工介入（取消订单、调整优先级等） |

#### 1.2 服务器端

| 功能模块 | 说明 |
|---|---|
| 订单管理 | 接收订单、维护订单状态流转（出餐→排队→配送中→完成） |
| 调度引擎 | 根据优先级规则和同楼层批量优化策略，向Lumi下发配送任务 |
| 状态同步 | 实时接收Lumi上报的位置、电量、任务进度，并更新至界面 |
| 告警服务 | 接收Lumi异常上报，推送至运维人员，记录异常日志 |

---

### 二、机器人主程序开发

机器人主程序是Lumi的中枢控制逻辑，负责**任务状态管理**和**各执行机构的协调配合**。

#### 2.1 任务状态机

Lumi的任务生命周期通过状态机管理，核心状态如下：

```
待命 → 接收任务 → 前往取餐点 → 等待出餐确认
    → 前往电梯 → 乘梯至目标楼层 → 导航至房间门口
    → 视觉校准 → 放置咖啡 → 返回取餐点 → 待命
```

- 每个状态节点均包含**异常中断入口**（对应异常处理机制章节）
- 状态机支持暂停/恢复，便于异常处理后继续执行

#### 2.2 各执行机构协调

| 执行机构 | 职责 | 协调要点 |
|---|---|---|
| AGV底盘 | 移动导航、路径规划、避障 | 主程序下发目标坐标，底盘执行移动并反馈实时位姿 |
| 机械臂 | 取放咖啡杯（从托盘到放置点） | 主程序在到达目标位置后触发机械臂动作序列 |
| 视觉相机 | 识别放置位置标记，提供偏差数据 | 到达房间后主程序调用视觉模块进行纠偏，输出补偿坐标给机械臂 |
| 夹爪 | 夹取/释放咖啡杯 | 与机械臂动作时序联动，确保抓取和放置时夹爪状态正确 |

#### 2.3 关键控制逻辑

- **动作序列编排**：到达房间后，按顺序执行「停车稳定→视觉拍照→计算补偿→机械臂伸出→夹爪释放→机械臂收回→确认完成」
- **托盘管理**：记录托盘各槽位是否有咖啡，支持批量配送时按序放置
- **安全约束**：机械臂和夹爪动作前必须确认底盘已停止，禁止运动中操作

---

### 三、设备间通讯模块开发

各设备通过不同通讯协议交互，通讯模块负责封装底层协议，向上提供统一接口。

| 通讯链路 | 协议（待定） | 说明 |
|---|---|---|
| Lumi ↔ 调度服务器 | HTTP/MQTT | 接收任务指令、上报状态和异常 |
| Lumi ↔ 梯控系统 | TCP/HTTP | 呼叫电梯、查询电梯状态、发送楼层指令 |
| 主程序 ↔ AGV底盘 | TCP | 下发目标坐标、读取实时位姿和速度 |
| 主程序 ↔ 机械臂 | SDK | 下发运动指令、读取关节状态 |
| 主程序 ↔ 视觉相机 | USB | 触发拍照、获取图像或坐标结果 |
| 主程序 ↔ 夹爪 | 串口/IO | 发送开合指令、读取夹爪状态 |

#### 通讯模块设计要求

- 每条通讯链路独立封装，包含**连接管理、数据收发、心跳保活、断线重连**
- 关键指令需支持**应答确认**机制，超时未应答则触发异常处理流程
- 所有通讯数据记录日志，便于故障排查
